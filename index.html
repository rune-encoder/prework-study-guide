Vocabulary

  • User Stories: Tasks you need to complete.
    1. Identify user of website created. (meets user's needs).
    2. Short description of the user you're creating an application for. (keep audience in mind).

  • Acceptance Criteria: Checklist which ensures you completed an application.
    1. Know when the application is finished
    2. A list of things or applications you must achieve to be considered "done".

  • Computational Thinking: Understand complex problems by breaking into smaller parts, developing solutions that can be clearly presented. Something the Computer/Human understands.
  • Pseudocode: Plain-Language description of what our code will do. Helps organize code.
  • Code Snippet: A portion of code.
  • Decomposition: To breakdown a problem into smaller tasks.
  • Pattern Recognition: Thinking about the how we've solved these subtasks.
  • Algorithm: A sequence of steps or rules that we can use to solve a problem.
  • Abstraction: Focusing on relevant information and disregard details that will not help.
  • Event Trigger: The completion of an event that will trigger a "second" event.
  • Conditional Statement: Allows us to make decisions based on conditions of oure code.
  • Debugging: Changing our original algorithm after finding a mistake in our pseudocode.
  • Loop: An action that we will repeat over and over again until a condition is met.

  • Agile Software Development: (Agile) helps developers handle change and uncertainty during the development process when issues arise. (Discover opportunity to enhance algorithm).
  • Pair Programming: (A practice) Occurs when two developers collaborate on a feature or a project. Can be in person or remote.
  • Driver-Navigator Model: (Pair Programming) The developer using the keyboard to type code "Driver". The other developer "Navigator" is responsible for the direction the pair wants to go. The two developers will switch roles every now and then (can be a couple of minutes).

  • Package: contains code written by others that we can add to and use on our computers and in our projects.
  • Package Manager: Automates the process of installing, upgrading, configuring, and removing packages. (ex. Homebrew).
  • Command-Line Interface (CLI): A command line program that uses text commands to communicate with the computers Shell.
  • Graphical User Interface (GUI): A User-friendly interface. (ex. Icons, Menus, Ex.).

  • Terminal: An application that allows us to interact with Shell.
  • Shell: A software that interprets and executes the various commands that we type in the terminal. (Note: Git-Bash or Bash Commands popular versions of shell.
  • SSH Key: is an access credential for the Secure Shell (SSH) Network Protocol. Generating SSH Keys helps developers to interface with certain remote services without having to constantly retype login information.
  • SSH Agent: A program that keeps track of keys and passphrases.

  • Git Bash: Allows users to use standard Unix commands that would otherwise not be available to a windows user.
  • Git Hub: A cloud based service that allows developers to store, manager and collaborate on code. It is conected to git.
  • Git: A software on your computer which tracks the changes that you make to your code. (version control).
  • Commits: Changes to your code. Operation which sends the latests changes of the source code to a repository.

  • Web Development Project: is a web application (web app) that provides a solution to a real-world problem that we want to solve.

  • Web Deployment Process:
    1. Identify the problem: Research is conducted to determine how the application will solve a problem for the intended audience.
    2. End Users: Intended audience of an application.
    3. Plan and Gather Requirements: The budget and resource needs are determined.
    4. Design: Details and requirements for the application are documented.
    5. Develop: Code begins and continues.
    6. Test: Issues are documented and fixed and application is tested until it reaches a standard for the public.
    7. Deploy: The application is published and made available to end users.
    8. Maintain: Run tests and write code to iimprove performance and features.

  • Web Development Methodologies:
    1. Waterfall: All requirements are collected and then each step is completed sequentially one at a time. This model si better suited to smaller projects.
    2. Agile: The project is broken down into small components which are built on an incremental basis. Feedback and testing are continually part of the process to improve and deliver an application that functions well and solves the problem.

  • Minimum Viable Product (MVP): The first iteration of a product without extra features. After completion, development may move to a different more robust version. More user friendly and functional. The first version will fulfil the requirements and proves that the solution to the program works.
  • Verson Control: Is the practive of tracking changes to our code. If we make a change to our code and decide that we don't want to keep them or they have a bug we can easily revert to an earlier version of code.
  • SSH or Hash: A unique identifier assigned when we save a version of our work using version control.

  • Markdown: is a lightweight plain-text formatting language. (ex. README).

  • Repositories (Repos): Where all files and version history are stored. Folders to store the following:
    1. Files that contain code.
    2. Files that contain other information about code. (ex. README.txt).
    3. The version history for all the files in the repository.
    4. Note: Can be owned by 1 developer or shared by a team.

  • Issues: Made for the repository so that we can keep track key tasks and progress in a project.
    1. A built-in to-do list.
    2. Track, organize and communicate ideas, tasks, bugs, and enhancements.
    3. Identify bottlenecks or blocks that impede progress.
    4. Avoid duplicating work by assigning tasks or issues.

  • Blocker: You are stuck on a particular problem that is preventing you from moving forward.
  • Bottleneck: A point in our development process that is slowing down the rest of the production.
  • Feature: A functional part of the application or website that has value or fulfills a need in the business.
  • Local Environment: A workspace where each developer does their work independently on their own computer.
  • Integrated Development Environment (IDE): A type of software used by developers that combines many of the features needed by developers to do their work in a single interface (GUI). Contains special tools needed for code development. (ex. Source Code Editor, Local Build Automation, Debugger, Visual Code Studio).
  • Code Editor: Text editor made for writing software that contains the tool necessary for syntax, highlighting and executing the code.

  • Default (Main) Branch: Will hold the final product of a project.
  • Feature Branch: What is currently being developed for a single feature in the application. Working Branch makes commits of all changes made.
  • Develop Branch: Can hold the code for multiple feature branches that have not been incorporated into the main branch yet.
  • Pull Request (PR): Is used to merge a comparison branch (in our case the feature branch) into the base branch (main branch).
  • Commit Changes: A saved record of a change made to a file within a repository.
  • Branch: A workspace in which you an make changes that wont affect the live site.
  • Merge: After a pull requests is approved the comit will be pulled in (or merged) from one branch to another and then deployed to a live site.

  • Federalist: A platform that securely deploys a website from a github repository in minutes and lets users preview proposed published changes.

HTML

  • head element: contains information about the webpage.
  • body element: represents the visible content shown to the user.
  • title element: defines the title for a webpage. *important for SEO*
  • style element: defines the CSS styles associated with the website.
  • base element: defines the base URL for a webpage
  • link element: connects external source to the HTML
  • meta element: defines metadata such as character set, description, keywords, authors and viewpoints.
  • script element: embeds client-side scripts in the HTML document.
  • *UTF-8 - allows us to use most characters & symbols on the page.
  • charset element: sets our character encoding.
  • http-equiv element: sets an HTTP header for content.
  • viewport element: sets the default view for webpage.
  • h1, h2, h3, h4, h5, h6 elements: header - level of heading a text block represents.
  • p element: paragraph or block of text.
  • ul element: unordered list. (bulleted list)

  • br element: creates a line of empty space or a break between two blocks of content.
  • a element: (anchor) - creates links to the same webpage or other webpages.

  • img element: contains information about images that are displayed on the webpage.
  • src attribute: defines the location or source of image file.
  • alt attribute: text string that describes the appearance and functionality of image.
  • main element: -?-

  • id attribute: HTML uses the id as a way to identify elements and select them.

CSS

  • inline CSS style sheet: inline styles apply to a specific HTML tag using a style attribute with a CSS ruel to style a page element ONLY.
  • internal CSS style sheet: contains CSS rules for the webpage in the head section of the HTML file. Ruel will apply to all tagged page elements.
  • external CSS style sheet: keeps all the CSS rules in a seperate file which makes design choices easier.
  • link element: used in HTML to link CSS style sheet. (used with external style sheet)
  • rel attribute: specifies the relationship between the current document and the linked document resource.
  • href attribute: specifies the location URL of the external resource. Uses relative patch / location derived from the current location.
  • example: (link rel="stylesheet" href=".assets/style.css")
  • seperation of concerns: seperating code into different files based on use. The idea that each section of code should have its own responsibility.
  • declarations: 1. CSS property we want to apply, 2. The value of the property.
  • margin property: indicates how much space we want around the OUTSIDE of an element.
  • padding property: indicates how much space we want around the content INSIDE an element.
  • CSS box model: HTML element is essentially a box. Each box consists of margins outside the element and the element's borders, padding, and content.
  • (*) selector: (add before declaration) wild card symbol to apply the rule to all the elemnts visible on the page.
  • do not repeat yourself (DRY): coding principle that encourages developers to reuse or share pieces of code in order to reduce the number of lines of code that need to be written or maintained.
  • text-align positioning property: way to center a title or header.
  • block value: assigns each image to start on a new line.
  • selector: defines the element or attributes to which the rules or declaractions will apply.
  • class attribute: allows us to share a css rule to any element we choose by assigning the rule to a class attribute with a class selector.
  • . preceding "classs" designates this as a class selector. ex. .card - HTML assigned to the class card will gain these properties from CSS.
  • example: (from HTML) (section class="card")

Git

  • Git Cycle:
    1. Create a "repository" project with a git hosting tool. (and issues)

    2. Copy (or clone) the repository to your local machine:
        3. git clone < paste copied SSH value >

    3. Add a file to your local repo and "commit" (save) the changes:
        4. git status: checks what branch we are currently on.
        git checkout -b "branch-name": creates a new branch and switches to it.
        "branch-name": prefix feature to indicate that it is a feature. name of new branch. ex. feature/starer-code
        (checkout): command is to move the working branch to a new branch.
        (-b): flag creates a new branch.
        Make Changes:
        git add -A: add the changed files in our working directory to the staging area of Git. (stage changes).
        (add): flag adds modifications in the current working brank to the staging area.
        (-A): flag indicates that we want to add all changes.
        git commit -m "description of change": once changes are staged, we can commit them to our repository. Once completed, our Git Repository is now up to date with our directory.
        (-m): flag used to associate a message with our commit. Short descriptive messages of change implemented.

    4. "Push" your changes to your main branch
        5. git pull origin main: pull the latest "main" branch. / make sure branch is up-to-date. *do before pushing code making sure branch is up to date*.
        (pull): command to receive a branch's modicfaction into the local environemnt.
        (origin): keyword indicates the branch. ex. main.
        (main): indicates the branch. main is from GitHub repo.
        git push origin "name-of-branch": push up all of our changes.
        (push): command 1. you can create a new remote branch. 2. add new changes to an existing branch.
        git checkout main: switches to main branch

    5. Make a change to your file with a git hosting tool and commit

    6. "Pull" the changes to your local machine
        7. git pull origin main: pull the latest "main" branch. / make sure branch is up-to-date. *do before pushing code making sure branch is up to date*.

    7. Create a "branch" (version), make a change, commit the change
        8. git status: checks what branch we are currently on.
        git checkout -b "branch-name": creates a new branch and switches to it.
        git add -A: (or git add .) add the changed files in our working directory to the staging area of Git. (stage changes).
        git commit -m "description of change": once changes are staged, we can commit them to our repository. Once completed, our Git Repository is now up to date with our directory.

    8. Open a "pull request" (propose changes to the main branch)
        9. git pull origin main: pull the latest "main" branch. / make sure branch is up-to-date. *do before pushing code making sure branch is up to date*.

    9. "Merge" your branch to the main branch
        10. git push origin "name-of-branch": push up all of our changes.
        git checkout main: switches to main branch

Javascript

  • variable: (keyword: var) a named container that allows us to store data in oure code.
  • (=) assignment operator: assigns specific data that we want stored in our variable.
  • ex. var "name" = "data stored"
  • string: includes more than 1 stored data. *CONTAINED IN ("quotation marks")*
  • (;) : every Javascript line of code ends in a (;)
  • console.log (method to the file): used to output a message to the web console by adding an ARGUMENT.
  • console (console.log): refers to the test environment that developers use to check out their code.
  • .log() (console.log): is a *METHOD* / set of instructions that can be executed by a computer. Will output what is added in the parentheses to the console.
  • (script src="./assets/script.js/file_location")(/script): used to link script.js file to index.html file. (linked with BODY ELEMENT)
  • control flow: is the order in which a computer executes code in a script. Javascript code runs from first statement down to the last statement.
  • (if) conditional statement: code will only execute IF the statement meets a condition or is true.
  • (else if) conditional statement: adds more than 1 condition on top of (if) statement.
  • Boolean: logical data type that can have only the values true or false.
  • truthy: (value that is true) the if statement will do whatever action we ask it to do in between the curly brackets.
  • falsy: (value that is not true) the computer will ignore whatever is in between the curly brackets.
  • ex. falsy values: 0, -0, On, "", null, undefined, NaN, fales.
  • conditional statements and loops: control the order in which code is executed or if it is een executed at all.
  • (===) strict equality operator: checks to see if two values are equal and returns a Boolean result true if they are equal and false if they are not.
  • array: a single variable that is used to hold a group of data (usually related data).
  • index: a number which identifies the data in the array. (starts with 0)
  • (for) loops: uses the predictable patter of indices to perform a task on all the items in an array by allowing a signle code block to be executed over and over.
  • statement: the starting point of the loop. - ex. var x = 0;
  • condition: the second statement - ex. x < name.length;
  • final statement: allows the array to iterate over each item. - ex. x++ or x+1
  • comment out code: makes code inactive without deleting it incase you need it later. Use (/* before) and (*/ after) a block of code.
  • function: a set of instructions that tells a computer how to perform a task. (does not automatically execute - only when we call it / reusuable code)
  • "()": a functions name is followed by a parentheses ex. function helloWorld()
  • "{}": code to be executed is written inside curly brackets.
  • randomize a selection: use math function to select random number. ex. Math.random()
